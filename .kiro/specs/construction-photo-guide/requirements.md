# 要件定義書

## 1. システム概要

定点撮影ガイドWebアプリ（construction-photo-guide）は、建設現場などを定期的に同じ構図で撮影するためのフロントエンド専用Webアプリケーションである。過去に撮影した参照写真をブラウザ上でアップロードし、その写真をカメラ映像に半透明で重ねて表示することで、ユーザーが同じ位置・同じ構図で再撮影できるよう支援する。バックエンド・データベース・サーバー保存は一切使用しない。

---

## 2. 開発目的

建設中の校舎を定期的に同じ位置から撮影し、建築過程を記録すること。毎回の撮影で構図がズレる問題を解消し、前回写真との比較を容易にする。

---

## 3. 解決したい課題

- 定点撮影のたびに構図がズレてしまい、比較写真として使いにくい
- 前回の撮影位置・アングルを記憶に頼っているため再現性が低い
- 専用機材（三脚・固定カメラ）なしに、スマートフォンだけで構図を揃えたい
- 撮影結果をその場でダウンロードして端末に保存したい

---

## 4. 想定ユーザー

- 建設現場の記録担当者（施工管理者、学校関係者など）
- スマートフォンを主な撮影端末として使用する非エンジニア
- 定期的に同じ場所を訪れて撮影を行う人

---

## 5. 利用シーン

1. 初回撮影時：現場で写真を撮影し、後日ブラウザでアップロードして目印を配置する
2. 再撮影時：アプリを開き、参照写真をアップロードし、カメラ映像に重ねながら構図を合わせて撮影する
3. 撮影後：撮影結果をブラウザからダウンロードして端末に保存する

---

## 6. スコープ

### 対象範囲

- ローカル画像のアップロードと表示
- 参照画像上への目印（マーカー）の配置・削除
- スマートフォン・PCブラウザでのカメラ起動（getUserMedia）
- 参照画像の半透明オーバーレイ表示
- オーバーレイ透明度の調整
- カメラ映像を見ながらの撮影（canvas経由）
- 撮影画像のブラウザダウンロード
- 三分割グリッド表示（任意）
- 目印の表示/非表示切替（任意）
- 背面カメラ優先起動（任意）
- 撮影結果確認モーダル（任意）
- ガイド付き/撮影結果のみ、ダウンロード選択（任意）

### 対象外

- バックエンドAPI・サーバー処理
- データベース・クラウドストレージへの保存
- ユーザー認証・ログイン機能
- 撮影履歴の永続保存
- 複数セッションをまたいだデータ共有
- PWA化（任意扱い、必須ではない）
- 動画撮影・録画機能

## 8. 非機能要件

### 8-1. ユーザビリティ

- THE App SHALL 主要操作（アップロード→目印配置→撮影→ダウンロード）を3タップ以内で到達できるよう設計する
- THE App SHALL スマートフォン画面（375px〜430px幅）で全ての操作が指1本で完結できるUIを提供する
- THE App SHALL ボタン・タップ領域の最小サイズを44×44px以上とする

### 8-2. 互換性

- THE App SHALL iPhone Safari（iOS 15以上）およびAndroid Chrome（Android 10以上）で動作する
- THE App SHALL getUserMedia APIを使用してカメラにアクセスする
- IF ブラウザがgetUserMedia APIをサポートしない場合、THEN THE App SHALL ユーザーに非対応メッセージを表示する

### 8-3. 目印座標の精度

- THE App SHALL 目印の座標を参照画像に対する相対座標（0.0〜1.0の正規化値）で保持する
- THE App SHALL 画面サイズや表示領域が変化しても、目印の相対位置が参照画像上の同じ位置に表示されることを保証する

### 8-4. パフォーマンス

- THE App SHALL 参照画像のオーバーレイ描画をカメラ映像のフレームレート（最低15fps）を維持しながら実行する
- THE App SHALL 画像アップロード後、2秒以内にプレビューを表示する

### 8-5. プライバシー・セキュリティ

- THE App SHALL 画像データをサーバーに送信しない
- THE App SHALL カメラ映像をサーバーに送信しない
- THE App SHALL 全ての処理をブラウザ内（クライアントサイド）で完結させる

## 11. 画面内で保持する情報

バックエンド・DBは使用しないため、全データはReactのstateまたはuseRefで管理する。ページリロードで消える。

| データ名 | 型 | 保持場所 | 説明 |
|----------|-----|----------|------|
| referenceImage | File \| null | React state | アップロードされた参照画像ファイル |
| referenceImageURL | string \| null | React state | 参照画像のObjectURL（表示用） |
| markers | Marker[] | React state | 目印の相対座標リスト |
| overlayOpacity | number | React state | オーバーレイ透明度（0.0〜1.0） |
| showMarkers | boolean | React state | 目印表示フラグ |
| showGrid | boolean | React state | グリッド表示フラグ |
| capturedImageBlob | Blob \| null | React state | 撮影結果画像（モーダル表示用） |
| cameraStream | MediaStream \| null | useRef | カメラ映像ストリーム |

### Markerオブジェクト定義

```typescript
type Marker = {
  id: string;       // ユニークID（crypto.randomUUID等）
  x: number;        // 参照画像幅に対する相対X座標（0.0〜1.0）
  y: number;        // 参照画像高さに対する相対Y座標（0.0〜1.0）
};
```

## 12. 技術的考慮事項

### 12-1. カメラ制御

- `navigator.mediaDevices.getUserMedia({ video: { facingMode: 'environment' } })` で背面カメラを優先起動する
- カメラ映像は `<video>` 要素にストリームをセットして表示する
- 画面離脱時（SCR-01へ戻る時）は `stream.getTracks().forEach(t => t.stop())` でカメラを停止する

### 12-2. オーバーレイ描画

- カメラ映像（`<video>`）の上に `<canvas>` を絶対配置し、参照画像・目印・グリッドを描画する
- `requestAnimationFrame` ループで毎フレーム canvas を再描画する
- 参照画像は `canvas.drawImage()` で描画し、`globalAlpha` で透明度を制御する

### 12-3. 目印の座標管理

- 目印配置時：クリック/タップ座標を参照画像の表示サイズで割り、相対座標（0.0〜1.0）に変換して保存する
- 目印描画時：相対座標にcanvasの表示サイズを掛けてピクセル座標に変換して描画する
- これにより、画面サイズ変更・向き変更時も目印位置がズレない

### 12-4. 撮影処理

- シャッター時に `canvas.toBlob()` または `canvas.toDataURL()` で画像データを取得する
- ダウンロードは `<a download>` 要素を動的生成してクリックする方式で実装する
- ダウンロード対象は「カメラ映像のみ（オーバーレイなし）」または「オーバーレイ込み」を選択可能にする（任意）

### 12-5. HEIC対応

- iPhoneのデフォルト撮影形式（HEIC）はブラウザによって表示できない場合がある
- `heic2any` 等のライブラリを使用してJPEGに変換するか、ユーザーにJPEG/PNG形式での保存を案内する

### 12-6. レスポンシブ対応

- カメラ映像とオーバーレイcanvasは同じサイズ・同じ位置に重なるよう `position: absolute` で配置する
- 画面の向き変更（portrait/landscape）に対応するため、`ResizeObserver` または `window.onresize` でcanvasサイズを再計算する

## 13. 想定リスク・制約

| リスク | 影響 | 対策 |
|--------|------|------|
| iOSのカメラ権限制限 | Safari でカメラが起動しない | HTTPS必須。権限拒否時はエラーメッセージを表示 |
| HEIC形式の非対応 | 参照画像が表示されない | heic2anyライブラリ導入またはユーザーへの案内 |
| 画面向き変更によるズレ | 目印・オーバーレイがズレる | 相対座標管理＋ResizeObserverで再計算 |
| カメラ映像とオーバーレイのサイズ不一致 | 構図が合わない | videoとcanvasを同一サイズに同期する処理を実装 |
| ブラウザのgetUserMedia非対応 | カメラが起動しない | 非対応ブラウザ検出時にメッセージ表示 |
| 大容量画像のメモリ使用 | ブラウザがクラッシュする | アップロード時に画像をリサイズ（最大2048px等）して使用 |
| ObjectURLのメモリリーク | メモリ使用量が増加 | 不要になったObjectURLはURL.revokeObjectURL()で解放 |

## 付録：次に作るべき画面設計・コンポーネント設計の要約

### 推奨コンポーネント構成

```
App
├── HomeScreen（SCR-01）
│   ├── ImageUploader          # ファイル選択・ObjectURL生成
│   ├── MarkerCanvas           # 参照画像表示＋目印配置・削除
│   │   └── MarkerDot          # 個別目印の表示
│   └── ActionBar              # 「撮影開始」ボタン、目印表示切替
│
└── CameraScreen（SCR-02）
    ├── CameraView             # <video>要素でカメラ映像表示
    ├── OverlayCanvas          # 参照画像・目印・グリッドをrAFで描画
    ├── OpacitySlider          # 透明度調整スライダー
    ├── GridToggle             # グリッド表示切替（任意）
    ├── ShutterButton          # 撮影実行
    └── ResultModal（任意）    # 撮影結果確認・ダウンロード選択
```

### 状態管理の方針

- グローバルstateはReact Context（または Zustand）で管理し、HomeScreen↔CameraScreen間でデータを共有する
- `referenceImage`・`markers`・`overlayOpacity` はContextで保持する
- `cameraStream` はCameraScreen内のuseRefで管理し、アンマウント時に必ず停止する

### 画面設計の優先事項

1. **MarkerCanvas**：参照画像の表示サイズに対する相対座標変換ロジックが最重要。`getBoundingClientRect()`でcanvas表示サイズを取得し、クリック座標を正規化する
2. **OverlayCanvas**：`requestAnimationFrame`ループ内で `video` → `canvas` → `referenceImage` → `markers` の順に描画する。`globalAlpha`で透明度を制御する
3. **カメラ停止処理**：`useEffect`のクリーンアップ関数で必ずストリームを停止する（iOSでカメラインジケーターが消えない問題を防ぐ）
4. **レスポンシブ**：`<video>`と`<canvas>`は同一の親要素内に`position: absolute`で重ねる。親要素を`aspect-ratio: 16/9`または`aspect-ratio: 4/3`で固定するとズレが起きにくい
